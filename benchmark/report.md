# Benchmark Report

Generated from measured runs, not placeholders.

## Methodology
- Hardware/environment: AMD EPYC-Rome Processor on Linux-6.8.0-106-generic-x86_64-with-glibc2.39 (generated 2026-04-04 07:17:15Z).
- Data sources: `AAPL` uses the official 1-level LOBSTER sample message file downloaded into `data/`; `MSFT_synthetic`, `TSLA_synthetic`, and `SPY_synthetic` are synthetic LOBSTER-format message streams generated by `scripts/generate_synthetic_data.py`.
- Synthetic files are explicitly synthetic and are intended for comparative benchmarking only, not market microstructure inference.
- AAPL: 118,497 parsed messages and ~2,050 peak active price levels.
- MSFT_synthetic: 50,000 parsed messages and ~59 peak active price levels.
- TSLA_synthetic: 100,000 parsed messages and ~137 peak active price levels.
- SPY_synthetic: 200,000 parsed messages and ~216 peak active price levels.
- Measurement approach: 5 measured iterations after 1 warmup iteration(s), wall-clock throughput from the full file-processing run, and message latency percentiles from the mutation+analytics hot path.

## Analysis

- `flat_vec` was faster on: AAPL (+1.18% at ~2050 levels), MSFT_synthetic (+0.99% at ~59 levels), TSLA_synthetic (+0.32% at ~137 levels), SPY_synthetic (+2.37% at ~216 levels).
- The scaling pattern matches the expected trade-off: `std::map` keeps `O(log N)` insert/find/erase costs at the expense of pointer-heavy, cache-unfriendly traversal, while `flat_vec` pays `O(N)` for inserts/erasures but benefits from contiguous storage and lower cache miss rates.
- No crossover was observed in this run.
- In this codebase and on this hardware, the expected deep-book `std::map` crossover did not materialize; `flat_vec` still held a small throughput edge on the deepest measured workload.
- Approximate in-process memory usage favored `flat_vec`; on average, the `std::map` book footprint was about 1.01x larger across the measured matrix.
- The `std::map` runs also showed the expected fragmentation tendency: more node overhead and less predictable locality once active level counts and live order counts increased.

## Recommendation

- The measured dataset did not reveal a crossover where `std::map` overtook `flat_vec`, so `flat_vec` is the current default recommendation.
- That recommendation is based on the actual benchmark output here, not on the generic `O(log N)` vs `O(N)` expectation: even the deepest measured AAPL sample favored `flat_vec`.
